How Connectors Work

What is a Connector?

A connector is a link between two Rulezet instances. It stores the remote instance URL, optional authentication credentials, sync preferences, and a record of every pull that was ever triggered. Once created, any admin can trigger a pull to import detection rules (and optionally bundles) from that remote instance into the local one.

The connection is one-directional: this instance pulls from the remote. The remote instance does not need to know about this connector, and no data is ever pushed to it.

Setting up a Connector

Creating a connector requires an admin account. The form accepts the following fields:

Field Description
Name Display name shown in the connector table and in job logs.
Instance URL Base URL of the remote Rulezet instance — e.g. https://rulezet.org.
Trailing slashes are stripped automatically.
API Key Optional. If the remote requires authentication, enter its API key. It will be sent as X-API-KEY on every request.
Sync rules Enable to pull detection rules (YARA, Sigma, etc.) from the remote.
Sync bundles Enable to also pull public rule bundles. Rules are always pulled first so bundle membership can be resolved.
Ownership mode Shadow user — a dedicated ghost account is created to own all imported content. Your account stays clean.
Become owner — you (the triggering admin) own all imported rules.
After creation, use the Test Connection button to verify the remote is reachable and supports the sync API. The test also fetches public rule/bundle counts shown in the connector card.

The Sync API

Every Rulezet instance exposes a set of read-only federation endpoints under /api/sync/. These endpoints require no authentication by default (an API key can be enforced by the remote admin). They are what the connector queries during a pull.

GET
/api/sync/manifest

Returns the remote instance identity (name, version) and its capabilities (sync_rules, sync_bundles). Used as a preflight check — if this returns 404, the remote is running an older version that does not support federation.

GET
/api/sync/stats

Returns the total number of active rules and public bundles on the remote. Used to populate the connector info strip in the pull modal.

GET
/api/sync/rules?since=1970-01-01&page=1&per_page=500

Paginated list of non-deleted rules. Each rule payload includes its UUID, format, content, metadata, all tags, CVE IDs, and the full update history. The UUID exported is always the canonical origin UUID (remote_rule_uuid or uuid) so identity stays stable across federation hops. Up to 500 rules per page.

GET
/api/sync/bundles?since=1970-01-01&page=1&per_page=500

Paginated list of public bundles. Each bundle payload includes the list of canonical rule UUIDs it contains, so the pulling instance can rebuild exact bundle membership.

Pull lifecycle

When an admin clicks Pull, a background job of type connector_pull is queued. The job runs asynchronously in the background worker thread and reports progress in real time via the Jobs panel.

1
Preflight — manifest check
The job fetches /api/sync/manifest. If the endpoint returns 404, the pull fails immediately with a clear error message. Otherwise the remote version and capabilities are logged.
2
Pre-flight totals
A single one-item request is sent to /api/sync/rules and /api/sync/bundles to read the total field and initialise the progress bar accurately.
3
Paginated rule fetch & upsert
Rules are fetched 500 at a time (always from since=1970 — full history). For each rule, _upsert_rule() is called and the result is tallied as created / updated / skipped / error. After each page the progress counter is committed so the UI updates in real time.
4
Paginated bundle fetch & upsert (if enabled)
After all rules are stored, bundles are fetched and processed via _upsert_bundle(). Rules are always pulled first so bundle membership can be resolved by UUID.
5
Completion
A connector.pull_done activity log entry is written with full stats (created / updated / skipped / errors). last_sync_at is updated on the connector.

Pull logic (upsert)

There is a single pull mode. For each rule received from the remote, the same decision tree is always applied. Matching is done exclusively by UUID — never by content.

Decision tree — applied to every rule from the remote
Is there a local rule whose remote_rule_uuid or uuid matches the remote UUID?
Yes — a local match was found
If the matched rule is in the trash: it is automatically restored (soft-delete fields are cleared). Processing then continues with the update checks below.
If the content (to_string) differs from the remote: the old content is archived in RuleUpdateHistory before being overwritten. The local rule is updated in place — its id, votes, comments, and favorites are never touched.
Updated + history archived
If only metadata changed (title, description, author, version, license, source) with no content change: fields are updated silently, no history entry is created.
Updated (metadata only)
If content and metadata are identical: nothing is changed. Tags and CVEs are still re-synced (additive).
Skipped
No match found

A new local rule is created from the remote data. Tags, CVEs, and the full update history are imported from the remote.

Created
Tags and CVEs are always re-synced on every pull, even when content and metadata are identical. This ensures tags added upstream are reflected locally without requiring a content change.

Rule deduplication

Rules are matched exclusively by UUID — never by content.

Canonical UUID

When a rule is pulled, its remote_rule_uuid is set to the UUID exported by the remote (remote_rule_uuid or uuid on the remote side). Identity is preserved across multi-hop federations: if instance A pulls from B which pulled from C, the canonical UUID traces back to C.

Trashed rules are checked too

The local match query explicitly includes soft-deleted rules (active rules are preferred). This prevents a rule you deleted locally from being re-imported as a duplicate. If it is matched, it is restored and updated in place.

Own rules coming back

If a rule that originated on this instance was shared upstream and then comes back during a pull, the lookup also checks Rule.uuid == remote_uuid. This prevents your own rules from being duplicated as a new import.

Original source preserved

The source field (e.g. a GitHub URL) is copied from the remote and never overwritten with the connector URL. The connector URL is stored separately in sync_instance_url and displayed in the rule detail page as "Synced from".

Tag synchronization

Tags are re-synced on every pull, even when the rule content and metadata are identical. The sync is additive only — tags are never removed from a local rule, even if they were removed on the remote.

1
Exact name lookup
For each tag name from the remote, the local DB is searched by exact name (case-insensitive). If found and the tag is inactive, it is reactivated automatically.
2
Galaxy auto-install
If the tag name starts with misp-galaxy: and is not found locally, the corresponding MISP galaxy is auto-installed from the local submodule (once per galaxy type per pull). The lookup is then retried.
3
Galaxy stub creation
If the galaxy cluster still does not exist after auto-install (e.g. the local submodule is behind the remote), a stub tag is created so the rule can still be tagged correctly. Stubs are marked with galaxy_meta.stub = true.
4
Missing tags reported
Non-galaxy tags (taxonomy or manual tags) that don't exist locally are collected and reported in the job summary as "missing tags". The connector panel lets you import these tag families on demand directly from the MISP submodule.

Bundle synchronization

Bundles are only pulled if Sync bundles is enabled on the connector. Bundle membership sync is additive: rules a local user added to the bundle are never removed.

Scenario Behaviour
New bundle A new local bundle is created. Its name, description, tags, and CVE identifiers are copied from the remote. Rules are linked by canonical UUID.
Existing bundle Missing rule associations are added (additive). Tags and CVEs are merged. Name/description are updated only if the remote version is newer (updated_at comparison).
Unchanged If no new rules, tags, or metadata changes are detected, the bundle is skipped.
Rules are always pulled before bundles in the same job run, so local rule records are guaranteed to exist when bundle membership is resolved.

Ownership modes

Every imported rule and bundle must be associated with a local user account. You choose the ownership strategy when creating the connector.

Shadow user Default

A ghost user account is automatically created for the connector (shadow_<uuid8>@connector.local, random unusable password). All imported rules and bundles are owned by this ghost account.

  • Your admin account stays clean.
  • Imported content is clearly labelled as coming from the connector.
  • The ghost account cannot log in.
Become owner

The admin who triggered the pull becomes the owner of all imported rules and bundles.

  • Useful when you want full editorial control over imported content.
  • Imported rules appear in your profile like locally created ones.

Self-sync prevention

To prevent an instance from pulling from itself (which would cause duplicate rules), the pull endpoint compares the full host:port netloc of the connector URL against request.host.

If the URLs match, the pull is blocked both client-side (the pull button is disabled with an orange "self" badge) and server-side (HTTP 400). This check compares the full netloc — 127.0.0.1:7009 and 127.0.0.1:7010 are considered different instances.

The official Rulezet connector (pointing to https://rulezet.org) is a system connector — it is created automatically at startup and cannot be modified or deleted.
